Тема: Фіналізація арбітражного двіжка «labyda» та запуск режиму тотального сканування (Production Release)
1. Виправлення 404 помилки в конекторі Myriad Markets (Критично)
* Контекст: Під час роботи в режимі scan_all двіжок успішно вигрібає ID маркетів через модуль дискавері, але паде з помилкою 404 ClientResponseError під час виклику методу get_orderbook.
* Причина (згідно з документацією Myriad API V2): Ендпоінт GET /markets/:id/orderbook при використанні числових market_id обов'язково вимагає передачі ID мережі (network_id) як query-параметра. Зараз двіжок його не передає.
* Вимоги до реалізації:
   1. Модернізувати метод get_orderbook у src/arbitrage_engine/connectors/myriad.py.
   2. Забезпечити обов'язкову передачу параметрів network_id (значення 56 для BNB Chain) та outcome (0 або 1 залежно від сторони, яку аналізує двіжок) у форматі query-строки:
   3. Python
url = f"{self.api_url}/markets/{market_id}/orderbook"
params = {
    "network_id": self.chain_id,  # Наприклад, 56
    "outcome": outcome_id         # 0 (Yes) або 1 (No)
}
response = await self.session.get(url, headers=self.headers, params=params)
   4.    5.    6. Переконатися, що за замовчуванням парситься правильна модель даних. Згідно з документацією, для отримання CLOB-даних ордербука (замість стандартного AMM) у запитах списків маркетів або котирувань використовується параметр trading_model=ob. Перевірити, чи враховано це в методах дискавері.
2. Впровадження ExecutionReport для обробки часткового виконання (Partial Fill)
* Контекст: Поточний інтерфейс конекторів повертає пропонований статус у форматі filled: bool. Для високочастотного арбітражу на реальні кошти цього недостатньо, оскільки при частковому заповненні ордера двіжок не знає точного об'єму залишку і не може коректно виконати протокол екстреного виходу (Emergency Unwind).
* Вимоги до реалізації:
   1. Розширити інтерфейс статусів ордерів у src/arbitrage_engine/models.py. Замість булевого значення повертати структурований об'єкт ExecutionReport (через dataclass або Pydantic).
   2. Об'єкт повинен містити:
      * status (Enum: OPEN, PARTIAL, FILLED, CANCELLED, EXPIRED)
      * amount_requested (початковий об'єм ордера)
      * amount_filled (фактично виконаний об'єм у стакані)
      * remaining_amount (залишок: requested - filled)
      * avg_price (середня ціна виконання з урахуванням проковзування)
   3. У модулі src/arbitrage_engine/execution.py налаштувати логіку: якщо ExecutionReport.status == PARTIAL, двіжок повинен моментально ініціювати Emergency Unwind на протилежній біржі суворо на суму remaining_amount, щоб повністю збалансувати дельту позиції та усунути ринковий ризик.
3. Чиста реалізація режиму scan_all (Без заглушок у конфігу)
* Контекст: Наразі валідатор validate_config у config.py вимагає обов'язкової наявності хоча б одного об'єкта в масиві markets із заповненими полями токенів або лімітів, що змушує використовувати штучні заглушки (на кшталт DUMMY) при увімкненому тотальному скануванні.
* Вимоги до реалізації:
   1. Якщо в корені config.json виставлено параметр "scan_all": true, валідатор повинен автоматично ігнорувати перевірку наявності ручних контрактів у масиві markets.
   2. Двіжок повинен штатно запускатися з порожнім масивом "markets": [], якщо активовано глобальний сканер. Фільтрація має відбуватися виключно за параметром "categories_to_scan": ["sports", "esports", "finance"].

* Контекст: При увімкненому режимі scan_all бот моніторить десятки або сотні маркетів одночасно. Полінг через REST API з інтервалом 250 мс гарантовано призведе до виходу за ліміти rate-limiting (200 запитів/сек для API-ключів) та отримання помилок 429.
* Вимоги до реалізації:
   1. Використовуючи надану документацію (розділ Real-time Updates / WebSockets), перевести метод watch_order_book у connectors/myriad.py на роботу через постійне WebSocket-з'єднання з сервером wss://ws.myriadprotocol.com/ws.
   2. Реалізувати anonymous-handshake через bidirectional JSON-протокол Centrifugo (відправка { connect: {}, id: 1 } -> отримання відповіді -> підписка на канали у форматі orderbook:{networkId}:{marketId}).
   3. Бот повинен динамічно підписуватись на потрібні канали orderbook:56:{market_id} на основі результатів, отриманих від модуля Discovery, та оновлювати локальний зріз склянки по дельта-змінах (changes), не навантажуючи REST API.
